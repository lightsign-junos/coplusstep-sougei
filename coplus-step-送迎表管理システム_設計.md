# コプラスステップ 送迎表管理システム 設計書

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| アプリ名 | コプラスステップ 送迎表管理システム |
| 対象施設 | コプラスステップ 昭和島教室（就労継続支援B型事業所） |
| 利用者 | 施設スタッフ（Googleログイン認証済みのみ） |
| 認証方式 | Google OAuth 2.0 ＋ 許可アドレスリスト管理 |
| 管理者 | JUNOS（許可アドレスの追加・削除が可能） |
| 対応端末 | PC・スマートフォン・タブレット（レスポンシブ対応） |
| データ保存 | Google スプレッドシート連携（Google Apps Script経由） |

## 2. 機能要件

### 2-0. 認証・アクセス管理

#### Googleログイン（OAuth 2.0）

- アプリ起動時にGoogleアカウントでのログインを要求
- ログイン後、そのアカウントのメールアドレスが**許可リストに含まれるか**をチェック
- 許可リストにないアドレスはログイン後にアクセス拒否画面を表示
- ログアウト機能あり
- ログイン状態はセッションで保持（ページリロードで再ログイン不要）

#### 許可アドレス管理（管理者画面）

- 管理者（JUNOS）のみアクセスできる `/admin` ページ
- 機能：
  - 許可アドレスの一覧表示
  - アドレスの追加（メールアドレスを入力して登録）
  - アドレスの削除
  - 管理者アドレス（自分自身）は削除不可
- 許可リストは Googleスプレッドシートの `allowed_users` シートで管理
  - GAS経由で読み書き
  - 列構成：`email` / `name`（任意）/ `added_at` / `is_admin`

#### 技術方式

- **Firebase Authentication**（Google プロバイダー）を使用
  - Vercel + Reactとの相性が良く、Googleログインが最も簡単に実装可能
  - 無料枠で十分対応可能
- ログイン後のアドレス照合はフロントエンド側でGASから許可リストを取得して判定
- `is_admin = true` のアカウントのみ `/admin` にアクセス可能（ルートガード）

### 2-1. 送迎ルート管理（コア機能）

- **行き（登所）のみ**管理（帰りは行きのルートを逆順に運用するため管理不要）
- 各ルートに**車両・運転手・添乗員**を紐づける
- 利用者の乗降順番をドラッグ&ドロップで並び替え可能
- 乗降場所は**自由に登録**（自宅・駅・グループホーム・路上の乗降しやすい場所など、場所の種別を問わない）
  - 場所名・住所・メモを登録
  - 利用者ごとに複数の乗降場所を登録可能（日や状況によって使い分け）

#### 車両構成

| 車両名 | 稼働状況 |
|--------|----------|
| ピンク | 常時稼働 |
| 青 | 常時稼働 |
| ヴェル（ヴェルファイア） | 補助稼働（ピンク・青で人数が足りない時のみ使用） |

- ヴェルは週単位のスケジュール作成時に「使用する / しない」をトグルで切り替え可能
- ヴェル非使用の週はルート表示・PDF出力から自動的に除外

#### 送迎表作成フロー（1日分）

1. **利用者を選択**：その日乗る利用者をチェックボックスで選択
2. **車両に振り分け**：ピンク・青（・ヴェル）に割り当て
3. **事業所到着時刻を設定**：施設に何時までに到着させるかを入力
4. **ラスト乗車者を設定**：最後に迎える人（事業所に一番近い人）を指定 → その人の乗車時刻が自動確定
5. **乗車順をドラッグ&ドロップで組む**：順番を並べると各利用者の迎え時刻をリアルタイムで逆算出力
6. **確定 → 保存**

#### 時刻自動計算の仕様

- **計算方向**：事業所到着時刻から**逆算**
- **移動時間の取得**：**OSRM**（Open Source Routing Machine）を使用・完全無料・APIキー不要
  - 住所→緯度経度の変換：**国土地理院 ジオコーディングAPI**（完全無料・日本住所特化）
  - 各停車地点間の所要時間を自動取得（OpenStreetMapベース、渋滞考慮なし）
  - 乗降に要する時間は一律 **3分** を加算（固定値、将来的に変更可能）
  - 計算例：`[事業所到着] ← 移動X分 ← [ラスト乗車] ← 移動Y分+3分 ← [前の人] ← …`
- **順番変更時**：ドラッグ&ドロップで順番を変えるたびに全時刻を再計算・即時反映
- 時刻の手動上書きも可能（自動計算値を変えたい場合）
- 時刻バリデーション：逆転チェックのみ（警告なし、参考表示）

### 2-2. 日別送迎表

- カレンダーから日付を選択して当日の送迎表を表示
- 表示内容：
  - 行き・帰り別のルート一覧
  - 各ルート：車両名・運転手・添乗員・利用者リスト（乗降順）・予定時刻
- 当日の**欠席登録**：対象児童をその日のルートから除外
- 当日の**ルート変更**：順番・担当スタッフをその日限りで上書き編集
- 変更内容は「当日変更フラグ」を立てて通常データと区別

### 2-3. 月別シフト管理

- 月カレンダー形式で各日の送迎パターンを管理
- 曜日ごとのデフォルトルートを設定し、日別に上書き可能
- 祝日・長期休暇期間の特別ルートを設定
- 利用者ごとの利用曜日・送迎有無を月単位で一覧表示

#### 週コピー機能（★運用負荷を大幅削減）

- 任意の週のメンバー構成・ルート・担当スタッフをまるごと翌週（または指定週）にコピー
- コピー後に差分だけ編集すればOK
- コピー元とコピー先の週を選んでワンクリックで適用
- 急な変更（欠席・スタッフ交代）はコピー後に当日変更機能で対応
- **想定運用**：毎週月曜に「先週をコピー」→ その週の変更点だけ修正 → 完成

### 2-4. 利用者マスタ管理

- 登録情報：利用者名・連絡先・送迎要否・備考
- 利用曜日のデフォルト設定
- 乗降場所の複数登録（場所名・住所・メモを自由に設定。例：自宅、〇〇駅北口、グループホームXX、など）
- 各乗降場所に「行き用 / 帰り用 / 両方」の区分を設定可能

### 2-5. スタッフ・車両マスタ管理

- スタッフ登録：氏名・役割（運転手 / 添乗員 / 両方可）・連絡先
- 車両登録：車両名・定員・ナンバー

### 2-6. PDF出力

- **週単位PDF**（現行フォーマットを踏襲・改善）
  - A4横向き、1週間分（月〜土）を1枚に収める
  - 縦軸：乗車順（1〜最大人数）、横軸：日付×車両
  - 各セル：利用者名・迎え時刻
  - ヘッダー行：車両名・運転手・添乗員・合計人数
  - ヴェル非稼働週は列を自動省略
- **日別PDF**（当日の詳細確認用）
  - 車両ごとに乗降順・時刻・場所を縦並びで表示
  - 当日変更・欠席箇所をハイライト
- ブラウザの印刷機能（`window.print()`）で出力

#### 時間バリデーション（現行の手入力ミスを防ぐ）
- 前の停車時刻より早い時刻を設定した場合に警告表示
- 同一ルート内で時刻の逆転がある場合にアラート
- 設定時間が「物理的に無理」な場合（移動時間を考慮した最低間隔チェック）を任意で設定可能

## 4. データ設計（Googleスプレッドシート構成）

### シート構成

| シート名 | 内容 |
|----------|------|
| `allowed_users` | ログイン許可アドレスリスト（管理者が管理） |
| `members` | 利用者マスタ |
| `member_locations` | 利用者ごとの乗降場所（複数登録可） |
| `staff` | スタッフマスタ |
| `vehicles` | 車両マスタ |
| `routes` | ルートマスタ（基本設定） |
| `route_stops` | 各ルートの停車順・児童割り当て |
| `daily_overrides` | 当日変更・欠席記録（日付付き） |
| `monthly_schedule` | 月別シフト（日付×ルートID） |

### 主要データ項目

**members シート**
```
id, name, phone, default_days（例: "月火水木"）,
send_flag（行き有無）, return_flag（帰り有無）, notes
```

**member_locations シート**（乗降場所マスタ）
```
id, member_id, name（場所名）, address, direction（go/return/both）, notes
```

**routes シート**
```
id, name, direction（go/return）, vehicle_id, driver_id, attendant_id,
departure_time, notes
```

**route_stops シート**
```
id, route_id, member_id, order, scheduled_time, location_id（member_locationsのid参照）
```

**daily_overrides シート**
```
id, date, type（absent/route_change）, route_id, child_id,
new_order, new_time, new_driver_id, new_attendant_id, notes, created_at
```

## 6. Google Apps Script連携仕様

### エンドポイント設計（REST風）

```
GET  ?action=getMembers          → 利用者マスタ取得
GET  ?action=getDailyRoute&date=  → 指定日の送迎表取得
POST ?action=saveOverride         → 当日変更・欠席を保存
POST ?action=saveRoute            → ルートマスタ更新
GET  ?action=getMonthlySchedule&year=&month= → 月別シフト取得
```

### GAS側の処理

- `doGet(e)` / `doPost(e)` でルーティング
- スプレッドシートIDは GAS のプロパティサービスで管理
- レスポンスは JSON 形式で返却
- CORS対応のためヘッダーを設定

## 7. UI・デザイン方針

### トーン

コプラスの既存ブランド方針に準拠しつつ、**業務ツールとして視認性・操作性を最優先**とする。現行の手書き・Excel表の「ダサくて見づらい」を解消することを明確なゴールとする。

- **カラー**：ホワイトベース、車両ごとに識別カラーを設定（例：ピンク車 → ピンク系、青車 → 青系）
- **フォント**：Noto Sans JP（日本語UIの読みやすさ重視）
- **レイアウト**：
  - デスクトップ：週単位の横並びビュー（現行フォーマットの良い部分を継承）
  - モバイル：日別縦スクロール、車両タブ切り替え
- **印刷時**：背景色を除去、モノクロ対応、文字サイズ10〜11pt

### 週ビューの改善方針（現行との差分）

| 現行の問題 | 新システムでの解決 |
|---|---|
| 時刻が手入力で矛盾が起きる | 時刻バリデーションで警告 |
| 車両・日付で列が多く見づらい | 車両タブ切り替えで分離 |
| 欠席・変更の反映が手動 | アプリ上で即時反映 |
| 印刷フォーマットが固定・手直し必要 | 自動生成PDF |

### 当日変更の視認性

- 変更・欠席のある行は**黄色ハイライト** + アイコン表示
- PDF出力時も変更箇所に注釈を付与
